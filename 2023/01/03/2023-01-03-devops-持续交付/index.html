devops-持续交付

发表于2023-01-03|更新于2023-03-26|devops
|阅读量:

简介

&emsp;&emsp;持续部署是一种技术领域的操作，从某处获取软件包，并按照预先设计的方案将其安装到计算节点上。

&emsp;&emsp;持续交付则是一个业务决策活动，是指新构建的特性交付给用户，用户就可以看到并使用它们。

涉及到的角色

  • 持续集成
    • 开发 + 测试
  • 敏捷
    • 产品 + 开发 + 测试
  • DevOps
    • 开发 + 测试 + 运维
  • 持续交付1.0
    • 产品 + 开发 + 测试 + 运维
  • 持续交付2.0
    • 业务 + 产品 + 开发 + 测试 + 运维

&emsp;&emsp;持续交付2.0强调以业务为导向，从一开始就将业务问题进行分解，减少浪费的同时，快速找到正确的业务前进方向，简称为双环模型。

&emsp;&emsp;持续交付2.0 的4个核心工作原则:

  • 坚持少做
  • 持续分解问题
  • 坚持快速反馈
  • 持续改进并衡量

&emsp;&emsp;精益思想的核心是开发新产品，先做出一个简单的原型最小化可行产品（MVP），为了快速得到用户的真实反馈，从每次实验的结果中学习，然后快速迭代，持续修正。

双环模型

第一环-探索环

&emsp;&emsp;探索环包含4个可持续循环步骤，分为如下

  • 提问
    • 定义问题，通过有针对性的提问，找出客户的具体需求，并找出具体需求后的原因，既具体需求后要解决的根本问题
  • 锚定
    • 既定义结果目标指示器，针对问题进行信息收集，经过分析，去除干扰信息，识别问题假设
  • 共创
    • 既共同探索和创造解决或验证该问题的多种具有可行性的解决方案
  • 精炼
    • 既对所有的可行实验方案进行选择，找到最小可行性解决方案

第二环-验证环

  • 构建
    • 根据非数字化描述，将最小可行性解决方案准确地转换成符合质量要求的软件包
  • 运行
    • 是指将达到质量要求的软件包部署到生产环境或交到用户手中，并使之为用户提供服务
  • 监测
    • 是指收集生产系统中产生的数据，对系统进行监控，确保其正常运行
  • 决策
    • 是指将收集到的数据信息与探索得出的对应目标进行对比分析

4个核心原则

  • 坚持少做
  • 持续分解问题
  • 坚持快速反馈
  • 持续改进并衡量

探索环

&emsp;&emsp;市场变化快，产品经常会因为潜在用户对原型的理解偏差导致当初的设计不再时应市场需求。

&emsp;&emsp;产品货服务开发中常见的风险假设

  • 用户假设
    • 我们提供的产品服务是针对某类潜在用户人群的需求的假设
  • 问题假设
    • 目标用户之所以有这种需求，是因为他们的却存在某些痛点需要解决的假设
  • 解决方案假设
    • 解决方案假设，既我们提供的解决方案可以解决这些通点或问题

&emsp;&emsp;上述三种假设，任何一个不成立都会导致我们事倍功半，甚至前功尽弃。

&emsp;&emsp;探索环的目标就是持续识别和定义这些有价值的假设，选择并验证风险最高或最易验证的价值假设。

提问

&emsp;&emsp;接到一个工作任务时，不要过早地进入解决方案讨论，先了解需求背后的真正原因忽略了对问题的讨论。

  • 问题域的提出方及解决方案的提供方代表尽量到场，参与讨论
  • 多问几个“为什么”
  • 在条件允许的情况下，尽可能收集数据信息，以便作为问题理解和分析的佐证
  • 移情，使用同理心。设身处地站在客户或问题提出方的角度思考问题，还原客户问题的场景

锚定

&emsp;&emsp;当选好要解决的问题领域后，接下来就要确定要达成的具体目标与结果。

&emsp;&emsp;避免模糊不清的目标，会影响团队成员之间的交流

&emsp;&emsp;例如服务用户数、产品收入等等功能。

工作原则

分解并快速试错

&emsp;&emsp;一次到位式解决方案通常具有较高的实施成本，带来的实际效果具有较高的不确定性。

&emsp;&emsp;如果换一种思路，更多使用低成本快速试验方式，那么在相同成本下，可以尝试更多的方法，能够尝试更多的次数，意味可能会有更多的收益。

一次只验证一点

&emsp;&emsp;一次只验证一个需求假设，在执行整个试验方案过程种，仍旧要保持开放心态，不断优化这些试验方案。

共创与精炼的常用方法

装饰窗方法

&emsp;&emsp;为新功能预留一个入口，让用户能看到，实际上没有实现其功能。

&emsp;&emsp;可以快速了解用户喜好的方法，并且成本低，以及紧急成都，为后续研发更全面的解决方案提供数据支持。

最小可行性特性法

&emsp;&emsp;决定开发一个新功能的时候，不要考虑的特别全面，先增加一个简易的功能验证用户是否喜欢该功能，如果喜欢继续推进，如果不喜欢可以直接停止。

&emsp;&emsp;前期可以采用比较笨拙的方式。

特区法

&emsp;&emsp;在特定用户范围内进行试验，以验证新功能的有效性，这样即使新功能无效或者效果不好，也不会影响区外的用户。

&emsp;&emsp;这种方法曾经用在共享单车和共享汽车上。

定向探索法

&emsp;&emsp;针对具有特定行为的特定用户群体，根据该用户具体行为模式，设计调查提纲，有针对性探索其行为背后的动机。

稻草人法

&emsp;&emsp;不开发任何真实的功能，只假装这个功能已经完成，并向用户展示该功能的真实效果，从而得到反馈。
